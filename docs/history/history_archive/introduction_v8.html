Chapter 1

設計哲學與痛點起源

為什麼傳統排障方式已無法應對現代製程複雜度？Glass Box 如何重新定義診斷體驗？

Pain Points

傳統排障的三大困境

現代半導體蝕刻製程的每一條控制迴路，都在每個製程批次（Lot）產生數以千計的感測器紀錄與 SPC 數據。當 OOC（Out-of-Control）警報觸發時，工程師面對的是：

🗄️ 資料孤島——數據在那裡，但無法快速存取

SPC 系統、APC（先進製程控制）、MES 配方管理、設備 EC 常數——每個系統都有自己的介面，查詢需要多次登入，資料無法交叉比對。工程師平均需要 30 分鐘以上才能收集齊所有診斷所需的原始資料。

🧠 經驗孤島——Know-how 在人腦裡，無法規模化

資深工程師憑藉多年累積的感覺與直覺進行判斷，但這些知識難以文字化，更難以傳承。新進人員面對相同警報時，往往需要反覆請教，造成大量隱性時間損失。

⏳ 時間孤島——診斷效率受限於人力資源

深夜班或假日班組警報升起時，可能沒有足夠的資深工程師可以立即回應。每延誤一小時，即可能影響數十個 Lot 的生產排程，造成直接的良率與交期損失。

Glass Box AI 診斷引擎 v8 主介面 — 左側為對話區，右側為診斷報告區，整合所有診斷功能於一個視窗

Agentic Workflow

Agentic Workflow — AI 主動驅動診斷

「Glass Box 不是讓工程師跟 AI 聊天——而是讓 AI 主動執行工程師的 SOP，並將結果以可追溯的方式呈現給人類做最終判斷。」

傳統 AI 輔助模式（問答式）

  • 工程師需自行收集資料
  • 將資料複製貼入 ChatGPT / AI 對話框
  • AI 提供建議，但無法驗證
  • 每次都要重複同樣步驟
  • 結論無可追溯的數據依據

Glass Box Agentic 模式

  • 事件觸發 → AI 自動調度技能
  • AI 主動查詢各系統資料（透過 MCP）
  • 每個診斷結論附帶真實數據表格
  • 技能可複用、可擴充、可維護
  • 工程師只需審閱結論 + 確認行動

AI 自動完成所有資料查詢與分析，報告卡片中包含真實 MCP 查詢資料表格，結論完全可追溯

Chapter 2

No-Code MCP Builder

如何讓沒有軟體工程背景的製程工程師，也能為自己的診斷場景建立 AI 數據查詢工具？

Data Subject

Data Subject 語意化資料登記

每一個診斷場景都需要特定的資料。Data Subject 是對「一類可查詢資料」的語意描述，它告訴系統：這個資料主題是什麼、有哪些欄位、代表什麼意義。

Data Subject 包含什麼

  • 名稱（Name）：英文識別符，如 etch_apc_params
  • 描述（Description）：這個資料主題的用途與業務場景說明
  • Schema 定義：欄位名稱、型別、語意描述（JSON 格式）
  • 資料來源標記：說明資料從哪個系統獲取（APC、MES、設備 API 等）

為什麼需要 Data Subject？

Data Subject 是橋接「業務語義」與「程式碼」的關鍵。有了它，AI 在 Auto Map 時才能理解每個欄位的含義，進行正確的語意匹配；工程師也能用自然語言描述需求，而不是記憶欄位名稱。

Builder 的 Data Subject 管理頁面 — 以卡片式顯示所有已登記的資料主題，並可一鍵預覽資料

MCP Script Builder

自然語言 → Python 沙箱腳本

工程師只需用中文或英文描述「我需要查詢什麼資料」，系統自動生成符合 MCP 格式的 Python 腳本，無需任何程式設計背景。

1

輸入需求描述

「查詢 APC 的 RF Power 在過去 24 小時的趨勢，若偏差超過 5% 標記為異常」

2

AI 自動生成腳本

系統根據描述與 Data Subject Schema，生成符合 MCP 輸出格式的 Python 腳本

3

Try Run 即時驗證

點擊 Try Run，在沙箱環境執行腳本，結果以表格即時顯示

MCP 腳本編輯器 — 左側自然語言描述，右側自動生成的 Python 腳本，下方即時顯示 Try Run 結果

Universal Data Viewer

Universal Data Viewer — 統一資料預覽

無論 MCP 腳本返回什麼格式的資料，Universal Data Viewer 都能即時以統一格式呈現，讓工程師在開發過程中立即看到資料的實際樣貌。

支援的顯示格式

  • 資料表格（預設）：以分頁表格顯示 dataset 前 15 筆記錄
  • Plotly 圖表：若 MCP 返回 chart_data，自動渲染互動式圖表
  • JSON 原始輸出：可折疊查看完整 JSON 結構
  • 執行狀態：耗時、狀態碼、錯誤訊息（若失敗）

MCP 輸出格式規範

{
  "dataset": [
    {"timestamp": "...",
     "value": 450.2,
     "status": "NORMAL"},
    ...
  ],
  "ui_render": {
    "type": "table",
    "chart_data": null
  }
}
💡

設計理念：MCP 的 dataset 不僅用於 AI 推論，也會直接嵌入診斷報告卡片，讓每個結論都有數據依據，完全可追溯。

Chapter 3

Skill 系統的領域知識注入

如何將資深工程師腦海中的 SOP 與判斷邏輯，轉化為可複用、可擴充的 AI 診斷技能？

Auto Map

語意自動映射（Semantic Auto-Map）

建立 Skill 時，最繁瑣的步驟之一是設定「MCP 輸出欄位」到「Skill 診斷輸入」的對應關係。Glass Box 的 Auto Map 功能使用 LLM 進行語意理解，自動完成這個步驟。

傳統手動映射的問題

  • 工程師需逐欄設定對應關係
  • 欄位名稱可能使用縮寫或特殊命名
  • 不同 MCP 工具的輸出格式各異
  • 映射錯誤不易在早期發現

Auto Map 做什麼

  • 分析 MCP 的 Data Subject Schema
  • 分析 Skill 的診斷提示詞需求
  • LLM 進行跨系統語意理解
  • 自動生成映射表 + 置信度分數
  • 異常映射用顏色標示，供人工確認

Auto Map 結果 — 自動生成的映射表，每個映射項目包含置信度分數與語意說明

Diagnostic Prompting

診斷提示詞設計（Diagnostic Prompting）

每個 Skill 的核心是一段精心設計的 AI 系統提示詞（System Prompt），它定義了 AI 應該如何解讀 MCP 資料、做出什麼樣的判斷、用什麼格式回答。

提示詞的四個要素

🎯

診斷目標

明確告訴 AI 這個 Skill 要診斷什麼異常、判斷什麼風險

📋

判斷標準

量化的閾值、比較基準、異常定義（例：偏差 >5% 為異常）

📊

資料解讀

如何解讀 MCP 返回的欄位（如 saturated 欄位代表什麼意義）

📝

輸出格式

結論摘要 + 狀態分類（NORMAL/ABNORMAL）+ 建議行動

🎓

最佳實踐：將資深工程師的判斷邏輯轉化為提示詞時，重點在於「當 X 指標超過 Y 閾值時，代表 Z 問題，建議採取 W 行動」這個結構。越具體的提示詞，診斷準確率越高。

SOP Encapsulation

人類 SOP 的封裝與傳承

Glass Box Skill 系統的本質，是將資深工程師的作業程序（SOP）封裝為可執行的 AI 技能單元。每個 Skill 都是一份數位化的製程知識資產。

📖

知識顯性化

將工程師大腦中隱性的判斷標準，轉化為明確的提示詞，讓知識可以被記錄、審閱與修正。

♻️

技能可複用

同一個 Skill 可被多個 Event 類型引用。APC 飽和檢查 Skill 可同時服務於不同的 SPC OOC 事件。

🔄

迭代改善

透過 Validate Logic 功能，定期審視 Skill 的判斷準確率，根據實際診斷結果持續優化提示詞。

Skill 詳情抽屜 — 展示完整的技能設定，包含診斷提示詞、MCP 工具組合與映射關係，代表一個完整封裝的製程 SOP

Progressive Streaming

漸進式 SSE 串流——讓等待有意義

v8 的最大技術革新之一：診斷結果不再是「等所有技能跑完才顯示」，而是採用 Server-Sent Events（SSE） 技術，每完成一個技能即立即渲染一張報告 Tab。

SSE 事件流詳解

start
診斷任務啟動，回傳事件基本資訊與技能總數（skill_count）
skill_start
某技能開始執行，前端 Chat 顯示「⏳ 正在執行 技能名稱...」
skill_done
某技能完成，攜帶完整診斷結果（狀態、結論、MCP 資料），前端立即新增 Tab
skill_start
下一個技能繼續執行...
skill_done
技能完成，再次新增 Tab
done
全部技能完成，渲染診斷總覽 Summary Bar，解鎖 UI
💻

技術實作：前端使用 fetch() + ReadableStream（而非 EventSource），以確保能在請求標頭中攜帶 JWT Authorization Token，突破 EventSource 無法設定 Header 的限制。

診斷串流進行中 — Chat 面板即時顯示各技能執行狀態，右側同步渲染完成的技能報告 Tab
