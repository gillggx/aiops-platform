Chapter 1

系統定位與核心價值

GlassBox AIoPs 將資深 PE 工程師的診斷知識固化為可配置的「診斷技能（Skill）」，由 AI 在製程異常發生時自動執行，大幅縮短根因分析時間。

Chapter 2

彈性四層組件架構

系統最核心的設計是四層組件金字塔。每一層都可獨立配置，PE 工程師無需修改程式碼，透過 Builder Copilot 的 UI 即可擴充新的診斷場景。

四層架構定義

⚡
Event Type
定義「什麼事件觸發診斷」，包含事件名稱與屬性 Schema。事件發生時攜帶具體的 params 值。
  • 一個 Event Type 可綁定多個 Skill（平行執行）
  • 範例：SPC_OOC_Etch_CD（蝕刻 CD SPC 超出管制界限）
🤖
Skill
定義「如何診斷」，綁定一個 MCP + param_mappings（事件屬性 → MCP 輸入映射）+ diagnostic_prompt（診斷邏輯）。
  • 同一個 MCP 可被多個 Skill 共用
  • param_mappings 支援 AI Auto-Map 自動生成
  • 範例：檢查SPC 是否連續異常
🔧
MCP
定義「如何取資料並前處理」，包含 input_params、output_schema，以及 AI 生成的 Python processing_script。
  • 腳本在 RestrictedPython 沙箱中安全執行
  • 支援四個建置步驟（選 DS → 撈樣本 → 加工意圖 → 試跑）
  • 範例：Query SPC data
🗄️
DataSubject
定義「資料在哪裡」，提供 API Endpoint URL、HTTP Method 及 Response Schema。
  • 可指向任意 REST API（工廠 MES、LIMS、SPC 系統）
  • MCP 執行時系統自動呼叫 DataSubject API
  • 範例：SPC_Chart_Data（/api/v1/mock/spc-chart）

Chapter 3

SPC OOC 診斷完整示範

從事件觸發到診斷完成，以 SPC OOC 蝕刻異常為例，示範系統的完整執行流程與 SSE 漸進式串流機制。

Step 1：觸發 SPC OOC 事件

事件攜帶完整的 params：equipment_id=TETCH01、chamber_id=PM2、measurement_value=102.5（超過 UCL=100.0）、spc_chart_id=CD_MEAN。這些屬性會透過各 Skill 的 param_mappings 傳遞給對應的 MCP。

Step 2：SSE 串流診斷執行（漸進式渲染）

點擊「啟動診斷分析」後，系統以 Server-Sent Events 串流方式逐 Skill 推播結果：

→ start 事件：診斷開始
右側報告區初始化，顯示事件 Header（事件型別、ID、params）
{"type":"start","skill_count":3,"event_type":"SPC_OOC_Etch_CD"}
→ skill_start × 3：各 Skill 開始
左側聊天區依序顯示「⏳ 正在執行 {Skill 名稱}...」
🤖 LLM → skill_done：每 Skill 完成立即推播
Claude 完成診斷分析後，右側立即追加對應的診斷卡片 Tab，不等待其他 Skill
{"type":"skill_done","index":0,"status":"ABNORMAL","diagnosis":"...","recommendation":"..."}
→ done：全部完成
渲染底部診斷摘要列，顯示各狀態計數
ℹ️ 為什麼使用 Fetch API 而非 EventSource？
瀏覽器的 EventSource API 不支援設定自定義 Header，無法傳遞 JWT Bearer Token。系統改用 Fetch API + ReadableStream，在設定 Authorization Header 的同時實現 SSE 串流。

Step 3：診斷結果 — 三個 Skill 的診斷卡片

「以 table 的方式回傳，呈現在用 Trend chart，x 軸是 datetime，y 軸是 UCL/LCL 用 orange 的顏色，Value 用綠色，但是超過 UCL/LCL 要用紅色的點」

Claude 生成的 Python 腳本（節錄）：

根據 DataSubject 的 Output Schema（datetime, value, UCL, LCL 等欄位）和使用者的可視化需求，Claude 自動生成符合 RestrictedPython 安全規則的資料處理腳本，包含：欄位提取、OOC 點標記、Trend chart 配色設定、輸出格式化。

💬 LLM 主動引導使用者修正加工意圖

若試跑結果不符合預期（如欄位缺失、圖表顯示錯誤），Claude 不只回傳錯誤訊息，而是直接分析原因並提示使用者如何修正加工意圖（Prompt）。例如：

  • 「您的意圖中未指定 UCL/LCL 欄位來源，請補充：y 軸 UCL/LCL 來自 schema 中的 UCL 和 LCL 欄位」
  • 「目前腳本回傳的 chart_data 是 HTML 字串，前端需要 Plotly JSON dict，請在意圖中加入：chart_data 需以 fig.to_dict() 格式回傳」

此設計讓 PE 工程師以對話式迭代精煉腳本，無需了解 Python 或 Plotly 細節。

System Prompt Key：PROMPT_MCP_GENERATE（在系統設定中可修改）

② MCP 試執行驗證（Step 2–4 完整流程）

②

試執行驗證 — 沙箱安全執行並驗證輸出格式

🤖 Claude Opus 4.6

點擊「⚡ 執行試跑」後，系統執行以下驗證迴圈：

  1. 使用 PROMPT_MCP_TRY_RUN 引導 Claude 根據加工意圖生成 Python 腳本
  2. 呼叫 DataSubject API（SPC_Chart_Data）取得樣本資料
  3. 在 RestrictedPython 沙箱中安全執行生成的腳本
  4. 比對輸出結構與 output_schema，回傳驗證報告

若驗證未通過，LLM 主動提示修正方向：

❌ 常見問題
腳本引用了 schema 中不存在的欄位 → Claude 提示：「請將欄位名稱改為 value（現有 schema 中無 measurement）」
chart_data 格式錯誤（HTML 而非 Plotly dict） → Claude 提示：「請在意圖末加入：以 fig.to_dict() 回傳圖表資料」
沙箱執行逾時 → Claude 提示：「資料量過大，建議在意圖中加入：僅取最近 100 筆資料」
✅ 成功後
狀態變更為 已儲存（快照），MCP 定義完成，可直接在 Skill Builder 中選用此 MCP

此迭代機制讓 PE 工程師以對話式 Prompt 修改取代傳統 Debug 流程，通常 1–2 次迭代即可完成腳本。

③④⑤ Skill Builder 的三個 AI 輔助功能

③

Auto-Map — 語意感知的事件屬性自動對應

🤖 Claude Opus 4.6

Skill 需要定義 param_mappings，指定事件的哪個屬性對應到 MCP 的哪個輸入參數。Auto-Map 讓 Claude 自動完成這個映射，並標示每個對應的信心度：

// Auto-Map 回傳範例
{
  "mcp_param": "chart_name",
  "event_field": "spc_chart_id",
  "confidence": "HIGH",
  "reasoning": "spc_chart_id 語意對應 chart_name，均為 SPC 管制圖識別符"
}

信心度（HIGH/MEDIUM/LOW）讓 PE 工程師快速識別需要人工確認的項目。

④

Suggest Logic — 資深 PE 等級的診斷邏輯建議

🤖 Claude Opus 4.6

Claude 以資深半導體製程工程師的角色，根據事件屬性 Schema 和 MCP output_schema，生成 3–5 條業界標準診斷邏輯，例如：

  • 「當 SPC 管制圖中連續 N 點超過 UCL/LCL，判定為趨勢性失控，需立即執行 PM 評估」
  • 「若量測值超出管制上限且偏移量 > 2σ，結合 APC 飽和度判斷是否為系統性漂移」
  • 「比較最近 24 小時的 recipe 修改紀錄與 OOC 發生時間，評估配方異動相關性」
⑤

Validate Logic — 防止診斷 Prompt 引用不存在欄位

🤖 Claude Opus 4.6

診斷 Prompt 中引用的欄位名稱（如 apc_saturation_level）必須存在於 MCP 的 output_schema，否則執行時會失敗。Validate Logic 讓 Claude 自動比對，回傳：

  • ✅ 所有欄位引用正確 — 可安全儲存
  • ❌ 欄位 xxx 不存在於 output_schema — 建議使用 yyy

Chapter 6

Skill — 連結事件、資料與 AI 診斷的核心組件

Skill 是 GlassBox AIoPs 的「智能核心」。它將 事件觸發時機、資料取得方式（MCP）、LLM 診斷邏輯 和 後續行動定義 整合為一個可配置的診斷單元，讓 PE 工程師的知識得以系統化複用。

Skill 的四個核心構成

⚡
Event 綁定
定義「什麼時候觸發這個 Skill」
  • 每個 Skill 綁定一個 Event Type（如 SPC_OOC_Etch_CD），當該事件發生時，系統自動啟動所有綁定至此 Event 的 Skill
  • 同一個 Event Type 可同時綁定多個 Skill（平行執行，每個從不同維度切入診斷）
  • param_mappings：定義事件攜帶的哪些參數值（如 lot_id、tool_id）要傳入此 Skill 使用的 MCP 查詢

範例：SPC_OOC_Etch_CD 事件同時觸發 APC 異常偵測、配方異動檢查、SPC 連續異常分析三個 Skill

🔧
MCP 連結
定義「要看哪些資料」
  • 每個 Skill 綁定一個 MCP，MCP 負責呼叫 DataSubject API 並執行資料前處理腳本
  • MCP 的輸出（dataset + ui_render）直接作為 LLM 診斷的輸入資料
  • SPC 案例中，MCP「Query SPC data」取回 100 筆量測資料，生成 Plotly Trend Chart（UCL/LCL 管制線 + OOC 紅點標記）和完整資料表格，供 LLM 分析與前端呈現
🤖
LLM 診斷
定義「如何判斷結論」
  • PE 工程師在 diagnostic_prompt 中以自然語言寫下判斷邏輯（如：「若連續 4 點超過 UCL，判定為 ABNORMAL」）
  • 執行時系統將 diagnostic_prompt + MCP 輸出資料（含量測時序、OOC 統計）組合為 LLM Context
  • Claude 分析後回傳結構化報告：status（NORMAL / ABNORMAL）、conclusion（診斷結論）、evidence（關鍵佐證清單）、summary（摘要）
  • Validate Logic 功能（LLM ⑤）可在建立前自動確認 diagnostic_prompt 中引用的欄位名稱均存在於 MCP output_schema，防止執行期錯誤
📋
後續行動
定義「診斷完成後操作員要做什麼」
  • human_recommendation 欄位由 PE 工程師事先定義，描述當此 Skill 判定為異常時操作員的標準處置流程
  • 自動呈現在診斷卡片底部「💡 建議行動」區塊，讓現場工程師無需查閱 SOP 手冊即知下一步
  • 範例：「立即停機確認 TETCH01 PM2 腔體狀態，聯繫設備工程師評估是否執行 PM 作業，並通知 APC 團隊確認補償模型更新頻率」

  • 資料表格：datetime / value / UCL / LCL / tool / lotID / recipe / OOC 旗標，一覽最近量測紀錄
LLM 可同時看到這份資料，用於在 diagnostic_prompt 的邏輯下分析連續超限情況，而操作員也能直觀看到圖表，快速理解 AI 診斷的依據。

Skill 的執行時序

① Event 觸發：系統接收到 SPC_OOC_Etch_CD 事件
查詢所有綁定至此 Event Type 的 Skill，本例有 3 個
② param_mappings 解析
每個 Skill 從事件 params（lot_id, tool_id, operation_number…）解析出自己需要的 MCP 輸入參數
③ DataSubject API 呼叫
各 Skill 依序（或可平行）呼叫 MCP 對應的 DataSubject endpoint，取得原始資料
④ MCP 腳本沙箱執行
RestrictedPython 沙箱執行 processing_script，輸出 Standard Payload（dataset + ui_render.chart_data）
🤖 ⑤ Claude LLM 診斷分析
diagnostic_prompt + MCP 輸出資料 → Claude 回傳 status / conclusion / evidence / summary JSON
⑥ SSE skill_done 推播，前端渲染卡片
卡片包含：狀態徽章（NORMAL / ABNORMAL）、AI 診斷結論、Evidence 清單、📊 Trend Chart、資料表格、💡 建議行動

🔮 Skill 設計的核心價值

Skill 的設計哲學是將「診斷知識」與「執行機制」分離：PE 工程師只需用自然語言定義「何時觸發、看什麼資料、如何判斷、建議做什麼」，系統自動處理 API 呼叫、腳本執行、LLM 呼叫、SSE 推播等技術細節。一個 PE 工程師定義的 Skill，可在日後無限次自動執行，讓資深知識從「人腦經驗」轉化為可量化、可稽核、可迭代的系統資產。
